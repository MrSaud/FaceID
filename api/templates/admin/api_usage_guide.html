{{ title }}

FaceID is a REST API for face detection, face comparison, and hardened liveness. Every /api/v1/ endpoint requires a client code and token created in this admin.

Quick start
  1. Go to API clients → Add.
  2. Save the client. Copy the code and token immediately — the token is shown only once.
  3. Send both values on every API request using the headers below.
  4. Test in browser at /test/liveness/.

1. Authentication

All requests to /api/v1/* must include:

X-Client-Code: CD82
Authorization: Bearer fid_live_xxxxxxxxxxxxxxxx
HeaderDescription
X-Client-CodeShort client code from admin, e.g. CD82
AuthorizationBearer + the secret token shown once on create/regenerate

Optional client restriction: set allowed_endpoints on the API client. Leave empty for full access, or use JSON like ["detect", "liveness", "compare", "health"].

2. Base URL

http://127.0.0.1:8000/api/v1/

3. Endpoints

MethodPathPurpose
GET/health/Service and model status
POST/detect/Detect face(s) in an image
POST/compare/Compare two faces
POST/liveness/challenge/Get a random liveness challenge
POST/liveness/Verify motion/action liveness
POST/liveness/speech/Verify say_digits with audio + video

4. Health check

curl -X GET http://127.0.0.1:8000/api/v1/health/ \
  -H "X-Client-Code: YOUR_CODE" \
  -H "Authorization: Bearer YOUR_TOKEN"

5. Detect face

JSON (base64)

curl -X POST http://127.0.0.1:8000/api/v1/detect/ \
  -H "X-Client-Code: YOUR_CODE" \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"image_base64": "BASE64_IMAGE_HERE"}'

Multipart file

curl -X POST http://127.0.0.1:8000/api/v1/detect/ \
  -H "X-Client-Code: YOUR_CODE" \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -F "image=@/path/to/photo.jpg"

6. Compare faces

curl -X POST http://127.0.0.1:8000/api/v1/compare/ \
  -H "X-Client-Code: YOUR_CODE" \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "image1_base64": "BASE64_IMAGE_1",
    "image2_base64": "BASE64_IMAGE_2",
    "threshold": 0.35
  }'

7. Liveness flow

Step A — create challenge

curl -X POST http://127.0.0.1:8000/api/v1/liveness/challenge/ \
  -H "X-Client-Code: YOUR_CODE" \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"digits_match_rate": 100}'

Random actions: blink, turn_left, turn_right, nod, smile, say_digits.

For say_digits, optional digits_match_rate:

  • 100 → 4 of 4 digits must match
  • 75 → 3 of 4
  • 50 → 2 of 4

Step B — motion challenges

For blink / turn / nod / smile, capture 5+ live camera frames while performing the action:

curl -X POST http://127.0.0.1:8000/api/v1/liveness/ \
  -H "X-Client-Code: YOUR_CODE" \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "challenge_id": "UUID_FROM_CHALLENGE",
    "challenge_token": "TOKEN_FROM_CHALLENGE",
    "source": "camera",
    "frames_base64": ["frame1", "frame2", "frame3", "frame4", "frame5"]
  }'

Step C — say_digits (speech)

Record camera + microphone together, then upload the clip:

curl -X POST http://127.0.0.1:8000/api/v1/liveness/speech/ \
  -H "X-Client-Code: YOUR_CODE" \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -F "recording=@/path/to/recording.webm" \
  -F "challenge_id=UUID_FROM_CHALLENGE" \
  -F "challenge_token=TOKEN_FROM_CHALLENGE" \
  -F "source=camera"

8. What liveness checks

  • Motion challenges: valid challenge, camera source, face motion, requested action, MiniFAS anti-spoof, screen-risk heuristic
  • say_digits: spoken digits match rate, lip-sync, same face throughout, motion, anti-spoof, screen risk

9. Risk score response

Liveness endpoints now return a confidence-based decision, not only is_live:

{
  "is_live": true,
  "risk_score": 0.18,
  "confidence": 0.82,
  "decision": "allow",
  "label": "live",
  "risk": {
    "risk_score": 0.18,
    "confidence": 0.82,
    "decision": "allow",
    "thresholds": { "allow": 0.30, "review": 0.60 },
    "signals": {
      "motion": 0.91,
      "action": 0.88,
      "antispoof": 0.76,
      "screen": 0.88
    }
  }
}
decisionMeaning
allowLow risk — treat as live (is_live: true)
reviewBorderline — retry, OTP, or manual review
denyHigh risk — treat as spoof (is_live: false)

Optional thresholds on liveness requests:

{
  "allow_risk": 0.30,
  "review_risk": 0.60
}

10. Common responses

StatusMeaning
200Request processed; check ok / is_live in body
400Bad image, recording, expired challenge, or validation error
403Missing/invalid code or token, expired client, or endpoint not allowed

11. Admin tips

  • Create one API client per external app or partner.
  • Use Regenerate token if a token is leaked.
  • Set expires_at for temporary access.
  • Disable clients with is_active = false instead of deleting them.
  • Check last_used_at to see recent API usage.
